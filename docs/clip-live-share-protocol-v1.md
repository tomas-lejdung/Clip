# Clip Live Share Protocol v1

## Scope

Clip Live Share Protocol v1 is the only signaling contract supported by Clip.
There is no legacy fallback. The Go service advertises rooms and routes bounded
opaque signaling envelopes; Clip owns admission, peer state, media, and viewer
count. After the initial WebRTC connection succeeds, media and control traffic
are peer-to-peer except when ICE requires a configured TURN relay.

## Trust boundary

The service may observe room names, network addresses, connection times,
temporary route identifiers, and envelope sizes. It must not receive plaintext
access codes, SDP, ICE candidates, stream metadata, codecs, media, or live
viewer state. It stores no session history.

The HTML viewer is trusted as part of the selected deployment. A server operator
who modifies that HTML is outside this protocol's threat model.

## Identifiers and limits

- Protocol identifier: `clip-live-share`
- Protocol version: `1`
- Room names: 3...64 ASCII characters, normalized to uppercase, containing only
  `A-Z`, `0-9`, and interior `-` characters.
- Owner token: 32 random bytes, base64url without padding. The server stores only
  SHA-256 of the decoded token.
- Room key agreement key: ephemeral P-256 key pair generated by Clip. Its private
  key remains in Clip memory. The X9.63 public representation is carried only in
  the viewer URL fragment as `#v=1&key=<base64url>`.
- Viewer key agreement key: ephemeral P-256 key pair per route. Its public X9.63
  representation is routed to Clip; its private key remains in the browser.
- Route identifiers: 16 random bytes, base64url without padding.
- Maximum pending routes per room: 8.
- Maximum connected viewers in Clip: 8.
- Maximum WebSocket frame: 262,144 bytes.
- Maximum decrypted inner message: 196,400 bytes. This leaves room for the
  AES-GCM tag, base64url expansion, route metadata, and JSON inside the
  262,144-byte WebSocket-frame ceiling.
- Maximum ICE candidates per peer and direction: 256.
- Initial answer timeout: 15 seconds.

## HTTP service

### Capabilities

`GET /.well-known/clip-live-share`

Returns public deployment configuration:

```json
{
  "protocol": "clip-live-share",
  "versions": [1],
  "serverVersion": "development",
  "viewerPathTemplate": "/{room}",
  "hostWebSocketPathTemplate": "/api/v1/rooms/{room}/host",
  "viewerWebSocketPathTemplate": "/api/v1/rooms/{room}/viewer",
  "iceServers": [
    { "urls": ["stun:stun.l.google.com:19302"] }
  ],
  "limits": {
    "maximumMessageBytes": 262144,
    "maximumPendingViewersPerRoom": 8
  }
}
```

### Advertise a room

`PUT /api/v1/rooms/{room}` with JSON:

```json
{ "ownerToken": "<base64url 32 random bytes>" }
```

Returns `201 Created` for a new room, `200 OK` for an idempotent request with
the same owner token, or `409 Conflict` when another host owns the name. The
response contains the normalized room and lease duration. The server keeps only
the token hash.

### Remove a room

`DELETE /api/v1/rooms/{room}` with `Authorization: Bearer <owner token>`.

### Utility endpoints

- `GET /healthz` returns process health only.
- `GET /version` returns build and protocol versions only.
- `GET /{room}` serves the embedded viewer for a syntactically valid room.

## WebSocket routing protocol

Clip connects to `/api/v1/rooms/{room}/host` with the owner-token Bearer header.
A browser connects to `/api/v1/rooms/{room}/viewer`. The host connection remains
open while the room is advertised. A viewer connection closes once its WebRTC
control DataChannel opens; the server therefore has no authoritative connected
viewer count.

JSON outer messages are deliberately small and contain routing metadata only.

### Viewer hello

Browser to server:

```json
{
  "type": "viewer-hello",
  "version": 1,
  "viewerKey": "<base64url P-256 X9.63 public key>"
}
```

Server to host:

```json
{
  "type": "route-opened",
  "routeId": "<random route identifier>",
  "viewerKey": "<unchanged viewer public key>"
}
```

Server to viewer:

```json
{
  "type": "route-opened",
  "routeId": "<the same random route identifier>"
}
```

Both endpoints must receive the same route identifier before deriving keys or
sending an encrypted relay envelope.

### Encrypted relay

Either endpoint sends:

```json
{
  "type": "relay",
  "routeId": "<route identifier>",
  "sequence": 1,
  "nonce": "<base64url 12 bytes>",
  "ciphertext": "<base64url AES-GCM ciphertext and 16-byte tag>"
}
```

The viewer route is implicit for browser-to-host frames. The server verifies
bounds and monotonic outer sequence numbers but never decrypts the ciphertext.
It forwards the envelope unchanged, adding only the route identifier when the
sender is a viewer.

Other outer messages are `route-closed`, `close-route`, `host-unavailable`, and
bounded protocol `error` messages. They never contain session content.

## Key derivation and encryption

Clip computes P-256 ECDH using its room private key and the viewer public key.
The viewer computes the same secret using its private key and the room public
key read from `location.hash`.

HKDF-SHA256 derives two 256-bit AES keys. The salt is SHA-256 of:

```text
clip-live-share|1|<normalized room>|<route id>
```

The HKDF info is `host-to-viewer` or `viewer-to-host`. Each encrypted direction
has an independent sequence beginning at one. Nonces are 12 cryptographically
random bytes. AES-GCM additional authenticated data is UTF-8:

```text
clip-live-share|1|<room>|<route id>|<direction>|<sequence>
```

Receivers reject duplicate, skipped, or decreasing sequence numbers, invalid
authentication tags, oversized plaintext, and a route mismatch. A fresh route
always performs fresh key agreement and resets its counters.

## Encrypted inner protocol

All inner messages contain `version: 1`, a `type`, and a per-route `sessionId`.

Admission occurs before allocating a native WebRTC peer:

1. Host sends `auth-challenge` with a fresh 32-byte challenge and whether an
   access code is required.
2. Viewer responds with `auth-response`. With an access code, `proof` is
   HMAC-SHA256 over the challenge and session identifier using SHA-256 of the
   normalized access code as the HMAC key. No access-code text is serialized.
3. Host compares the proof in constant time and responds with `auth-result`.
4. Only an allowed route may proceed to WebRTC negotiation.

Initial negotiation uses encrypted `offer`, `answer`, and `ice` messages with a
unique `negotiationId`. Once the browser observes the reliable ordered
`clip-control-v1` DataChannel open, it sends `close-route` and closes its viewer
signaling WebSocket. Clip treats that browser-initiated route closure as a
handoff and preserves the established peer. Later stream and codec renegotiation
travels over the DataChannel. A failed ICE transport opens a new route and
repeats encrypted signaling; optional encrypted resume tokens may be added
without changing the server routing contract.

## Media and control

- WebRTC DTLS-SRTP carries video and one optional Opus system-audio track.
- Stream identities are opaque random identifiers, never fixed `video0`...`video3`
  protocol names or SDP `mid` arithmetic.
- The first release retains Clip's four-window product limit, but the protocol
  and viewer manifest accept an arbitrary bounded stream list.
- `clip-control-v1` carries versioned authoritative manifests, stream activation,
  focus, geometry, cursor, sharing state, system-audio state, codec renegotiation,
  and session closure.
- Manifest `width` and `height` remain the encoded pixel dimensions. Current
  hosts also send the optional pair `sourcePointWidth` and `sourcePointHeight`
  so Retina captures retain their full pixel detail while viewers present them
  at the source Mac's logical size. Receivers fall back to encoded dimensions
  when both point fields are absent and reject an incomplete pair.
- A receiver echoes `session-closing` without a reason as an acknowledgement;
  the sender waits at most 500 ms before closing DTLS-SRTP.
- Clip counts viewers from connected WebRTC peer state. The server does not.

## Lifecycle

Rooms are in-memory leases. A connected host renews its lease. A transient host
disconnect keeps the advertisement reserved briefly for authenticated reconnect;
pending viewer routes close immediately. A process restart clears all rooms and
Clip re-advertises its current room. The initial deployment is intentionally a
single replica.
