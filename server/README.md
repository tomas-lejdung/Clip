# Clip Live Share Server

The Clip Live Share server advertises ephemeral room names, serves the trusted
browser viewer, and routes bounded encrypted signaling envelopes. It never
receives access codes, SDP, ICE candidates, stream metadata, media, or an
authoritative viewer count in plaintext.

The implementation follows [`docs/clip-live-share-protocol-v1.md`](../docs/clip-live-share-protocol-v1.md).

## Run locally

Go 1.25 or newer is required.

```bash
cd server
go run ./cmd/clip-live-share-server
```

The server listens on `:8080`. Configure Clip's Live Share endpoint as
`http://localhost:8080` while developing. A local viewer URL is
`http://localhost:8080/{ROOM}#v=1&key=...`; the fragment is generated by Clip
and is never sent to this service.

Useful checks:

```bash
curl http://localhost:8080/healthz
curl http://localhost:8080/version
curl http://localhost:8080/.well-known/clip-live-share
```

## Configuration

| Environment variable | Default | Meaning |
| --- | --- | --- |
| `PORT` | `8080` | Container-friendly listening port. |
| `CLIP_SERVER_ADDRESS` | `:8080` | Full listen address; takes precedence over `PORT`. |
| `CLIP_SERVER_LEASE_DURATION` | `5m` | Lifetime of an advertised room before a host connects. |
| `CLIP_SERVER_RECONNECT_GRACE` | `30s` | Time an authenticated host has to reclaim a disconnected room. |
| `CLIP_SERVER_ROUTE_IDLE_TIMEOUT` | `2m` | Maximum idle time for a pending signaling route. |
| `CLIP_SERVER_MAXIMUM_ROOMS` | `1024` | In-memory advertisement ceiling. |
| `CLIP_SERVER_MAXIMUM_CONNECTIONS` | `2048` | Host and pending-viewer WebSocket ceiling. |
| `CLIP_SERVER_RESERVED_HOST_CONNECTIONS` | `64` | Global connection slots viewers cannot consume. |
| `CLIP_SERVER_MAXIMUM_CONNECTIONS_PER_SOURCE` | `64` | Concurrent WebSocket ceiling per resolved client source. |
| `CLIP_SERVER_ROOM_ADVERTISEMENTS_PER_MINUTE` | `60` | Per-source room advertisement/renewal rate. |
| `CLIP_SERVER_WEBSOCKET_UPGRADES_PER_MINUTE` | `240` | Per-source host and viewer upgrade-attempt rate. |
| `CLIP_SERVER_MAXIMUM_QUEUED_BYTES_PER_SOCKET` | `2097152` | Per-socket queued and in-flight application-frame byte budget. |
| `CLIP_SERVER_MAXIMUM_QUEUED_BYTES_TOTAL` | `67108864` | Process-wide queued and in-flight application-frame byte budget. |
| `CLIP_SERVER_MAXIMUM_TRACKED_SOURCES` | `4096` | Ceiling for in-memory source admission records. |
| `CLIP_SERVER_TRUSTED_PROXY_CIDRS` | empty | Comma-separated proxy networks allowed to supply `X-Forwarded-For`. |
| `CLIP_SERVER_ALLOWED_ORIGINS` | same origin | Optional comma-separated additional browser origins. |
| `CLIP_SERVER_ICE_SERVERS_JSON` | Google STUN | Public WebRTC ICE server configuration returned by capabilities. |

Example ICE configuration:

```bash
export CLIP_SERVER_ICE_SERVERS_JSON='[{"urls":["stun:stun.example.com:3478"]}]'
```

Any TURN credential included in capabilities is delivered to viewers. Use
appropriately scoped credentials and do not place unrelated secrets in this
configuration.

Source throttles use the direct TCP peer by default and ignore forwarded
headers. Behind a TLS reverse proxy, either enforce client limits at that proxy
or set `CLIP_SERVER_TRUSTED_PROXY_CIDRS` to the proxy network. Only then does the
server resolve `X-Forwarded-For` from right to left, skipping trusted hops. Never
trust a network from which untrusted clients can connect directly.

## HTTP surface

- `GET /.well-known/clip-live-share` — protocol and deployment capabilities.
- `PUT /api/v1/rooms/{room}` — advertise a room using a client-generated owner capability.
- `DELETE /api/v1/rooms/{room}` — remove a room using the Bearer owner capability.
- `GET /api/v1/rooms/{room}/host` — authenticated host WebSocket.
- `GET /api/v1/rooms/{room}/viewer` — temporary viewer WebSocket.
- `GET /{room}` — embedded browser viewer.
- `GET /healthz` and `GET /version` — process-only status without room metrics.

All state is memory-only. Run one replica: a restart intentionally clears room
advertisements, and Clip re-advertises. TLS should terminate at a reverse proxy;
internet deployments must expose HTTPS/WSS rather than plaintext HTTP/WS.

monotonic relay sequences, strict message bounds, browser assets, security

## Docker

Build and run locally:

```bash
cd server
docker build --build-arg VERSION=development -t clip-live-share-server .
docker run --rm -p 8080:8080 clip-live-share-server
```

The runtime container is non-root and includes a `/healthz` health check.

Publish a multi-architecture Docker Hub release:

```bash
cd server
./scripts/publish-docker.sh 1.0.0
```

By default this publishes `linux/amd64` and `linux/arm64` tags to
`tomaslejdung/clip-live-share-server`. Override `DOCKER_REPOSITORY` or
`DOCKER_PLATFORMS` when needed.
